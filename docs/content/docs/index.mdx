Lohikeitto (Soup) is a service catalog API. It powers subscription search in Sharkie — users type a service name and get back logos, domains, and metadata from multiple sources.

Full endpoint documentation is available in the [API Reference](/api-reference).

## How search works

When a client sends `GET /search?q=spotify`, the API searches up to six sources **in parallel**:

```
           ┌─── Inhouse DB (PostgreSQL, pg_trgm) ── ~50ms
           │
 request ──┼─── App Store (iTunes Search API) ────── ~300ms
           │
           ├─── Google Play (HTML parsing) ────────── ~400ms
           │
           ├─── Web (OG / Twitter Card / favicon) ── ~500ms
           │
           ├─── Brandfetch API ───────────────────── ~500ms
           │
           └─── logo.dev API ─────────────────────── ~800ms
```

The `sources` parameter controls which sources are queried — as a comma-separated list:

```
/search?q=spotify&sources=inhouse,appstore,playstore
```

Aliases: `external` (brandfetch + logo.dev + appstore + playstore + web), `mobile` (appstore + playstore), `all` (default — all six).

### Regional & language parameters

By default, App Store and Google Play return results for the **US** store in **English**. You can override this per-store:

| Parameter | Applies to | Default | Example |
|-----------|-----------|---------|---------|
| `app_store_country` | App Store | `US` | `RU`, `JP`, `GB` |
| `playstore_country` | Google Play | `US` | `RU`, `JP`, `DE` |
| `language` | Google Play only | `en` | `ru`, `ja`, `de` |

All parameters are optional. Country codes are [ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2).

```
/search?q=сбербанк&sources=mobile&app_store_country=RU&playstore_country=RU&language=ru
```

Under the hood:
- **App Store** — country code in URL path (`https://itunes.apple.com/{country}/search`). The `language` parameter is ignored — iTunes localizes results based on the storefront region only.
- **Google Play** — `gl` (country) and `hl` (language) query params + `Accept-Language` header

Results are **deduplicated by domain** with priority:

```
inhouse  >  appstore  >  playstore  >  web  >  brandfetch  >  logo.dev
```

If a domain appears in multiple sources — only the highest-priority version is kept. Domains are **normalized** before comparison (`www.` prefix is stripped, case-insensitive), so `www.okcupid.com` and `okcupid.com` are treated as the same domain. If **any** domain of an external result is already claimed by a higher-priority source, the entire result is dropped.

Mobile store results use package/bundle IDs as their identifier (e.g. `com.spotify.client`), so they generally don't conflict with domain-based results. App Store results additionally include the seller's domain (from `sellerUrl`) as the **first** entry in `domains` — this enables dedup against inhouse services that share the same domain.

When 3+ sources return data, each external source is capped at **5 results** to avoid flooding. Inhouse results are not capped.

### Source details

| Source | Type | Search | Lookup | API key needed |
|--------|------|--------|--------|----------------|
| **inhouse** | PostgreSQL (pg_trgm) | Name, slug, domain, alternative names, bundle ID | UUID, slug, domain | No |
| **appstore** | iTunes Search API (JSON) | Keyword search | Exact by `bundleId` | No |
| **playstore** | Google Play (HTML parsing) | Search page parsing | Details page meta tags | No |
| **web** | Direct HTTP fetch | Domain-only (requires `.` in query) | Site icons, then OG/Twitter fallback | No |
| **brandfetch** | Brandfetch API (JSON) | Keyword search | Via search | `BRANDFETCH_CLIENT_ID` |
| **logo.dev** | logo.dev API (JSON) | Keyword search | Via search | `LOGODEV_PK`, `LOGODEV_SK` |

**Web source** fetches `https://{domain}` and extracts logos with priority: verified `/favicon.png` -> `apple-touch-icon` -> `<link rel="icon">` -> `og:image` / `twitter:image` -> `/favicon.ico`. The site name is taken from `og:site_name` / `og:title` / `<title>`.

**App Store** metadata includes `bundle_id`, `category_slug` (Apple genres mapped to local categories), and `tags`.

**Google Play** has no public JSON API — results are extracted by parsing HTML. Icon URLs are resized to 512px via the `=s512` suffix on `play-lh.googleusercontent.com`.

### Service lookup

`GET /service/{lookup}` resolves a service by UUID, slug, domain, or bundle ID. If not found locally, it searches external sources and stashes the result in the **limbus** (approval queue).

For mobile apps or web domains, use `?source_hint=` to trigger an exact lookup:

```
/service/org.getpure.pure-iphone?source_hint=appstore
/service/com.whatsapp?source_hint=playstore
/service/notion.so?source_hint=web
```

Without a hint, the lookup only searches **brandfetch + logo.dev** (fast, domain-based sources). Mobile stores and web parsing require an explicit hint.

When using `source_hint=appstore` or `source_hint=playstore`, you can specify the store region and language:

| Parameter | Applies to | Default | Example |
|-----------|-----------|---------|---------|
| `country` | appstore, playstore (requires `source_hint`) | `US` | `RU`, `JP` |
| `language` | playstore only (requires `source_hint=playstore`) | `en` | `ru`, `ja` |

```
/service/ru.sberbank.sberbankir?source_hint=appstore&country=RU
/service/com.whatsapp?source_hint=playstore&country=RU&language=ru
```

### Frontend strategy

The client fires two sequential requests to keep the UI responsive:

1. `GET /search?q=...&sources=inhouse` — renders instantly (~50ms)
2. `GET /search?q=...&sources=external` — appends results when ready (~1-2s)

External results use a **deterministic UUID v5** derived from the domain (or bundleId for App Store), so React keys stay stable across requests.

## Quick Start

```bash
cp lohikeitto.env.example lohikeitto.local.env
# fill in API keys (Brandfetch, logo.dev, Cloudflare R2)

make dev
```

Migrations run automatically on startup. The database is created if it doesn't exist.
